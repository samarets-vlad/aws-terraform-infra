# AWS Terraform Інфраструктура

[![Terraform CI](https://github.com/samarets-vlad/aws-terraform-infra/actions/workflows/terraform.yml/badge.svg)](https://github.com/samarets-vlad/aws-terraform-infra/actions/workflows/terraform.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)

> 🇬🇧 [English version](README.md)

Цей репозиторій містить готову до продакшену AWS інфраструктуру, описану у вигляді коду за допомогою **Terraform**.

Замість того, щоб вручну клікати в AWS Console для створення серверів, баз даних та балансувальників навантаження — все описано у файлах. Запускаєш одну команду — інфраструктура з'являється. Запускаєш іншу — зникає. Все відтворюється і зберігається в Git.

---

## Що створює ця інфраструктура?

Після виконання `terraform apply` в AWS акаунті створюються такі ресурси:

```
                        Інтернет
                           │
                    ┌──────▼──────┐
                    │     ALB     │  ← Балансувальник навантаження
                    │  (порт 80)  │    (розподіляє трафік)
                    └──────┬──────┘
                           │
              ┌────────────┴────────────┐
              │                         │
       ┌──────▼──────┐           ┌──────▼──────┐
       │   EC2 App   │           │   EC2 App   │  ← Сервери застосунку
       │  (AZ-1)     │           │  (AZ-2)     │    (2 зони доступності)
       └──────┬──────┘           └──────┬──────┘
              │                         │
              └────────────┬────────────┘
                           │
                    ┌──────▼──────┐
                    │  RDS (PG)   │  ← База даних PostgreSQL
                    └─────────────┘    (приватна, зашифрована)

       Всі ресурси знаходяться всередині VPC (Virtual Private Cloud)
       Публічна підмережа  → тільки ALB
       Приватна підмережа  → EC2 сервери застосунку
       DB підмережа        → RDS (без доступу до інтернету)
```

### Ресурси, що створюються:

| Ресурс | Що це таке | Примітки |
|--------|-----------|----------|
| **VPC** | Ізольована мережа в AWS | Твій власний приватний відділ хмари |
| **Публічні підмережі** | Підмережі з доступом до інтернету | Тут живе ALB |
| **Приватні підмережі** | Підмережі без доступу до інтернету | Тут живуть сервери застосунку |
| **DB підмережі** | Підмережі, ізольовані від усього | Тут живе база даних |
| **ALB** | Балансувальник навантаження | Приймає трафік користувачів, розподіляє по серверах |
| **EC2** | Віртуальні сервери | Запускають твій застосунок |
| **RDS (PostgreSQL)** | Керована база даних | Зашифрована, з автоматичними резервними копіями |
| **S3 bucket** | Файлове сховище | Для статичних файлів або файлів застосунку |
| **Security Groups** | Правила файрволу | Контролюють, хто з чим може спілкуватися |

## Структура репозиторію

```
aws-terraform-infra/
├── main.tf                    # Головна точка входу — з'єднує всі модулі
├── variables.tf               # Всі вхідні параметри з описами
├── outputs.tf                 # Що Terraform виводить після apply (URL, IP)
├── locals.tf                  # Внутрішні обчислені значення (спільні теги)
├── versions.tf                # Вимоги до версій Terraform та провайдера
├── backend.tf                 # Де зберігати state файл Terraform (S3)
├── providers.tf               # Конфігурація AWS провайдера
├── terraform.tfvars.example   # Приклад файлу зі змінними — скопіюй у terraform.tfvars
├── .tflint.hcl                # Конфігурація лінтера
├── .gitignore                 # Файли, які НЕ комітяться в Git (секрети, state тощо)
│
├── modules/                   # Багаторазові компоненти інфраструктури
│   ├── vpc/                   # VPC, підмережі, маршрутизація, інтернет-шлюз
│   ├── alb/                   # Балансувальник навантаження + target group
│   ├── ec2_app/               # EC2 інстанси + security group
│   └── rds/                   # RDS PostgreSQL + subnet group + security group
│
├── .github/
│   └── workflows/
│       └── terraform.yml      # CI/CD пайплайн (запускається при кожному push/PR)
│
├── SECURITY.md                # Політика безпеки та інструкції зі звітування
└── LICENSE                    # MIT ліцензія
```

**Що таке «модуль»?**  
Модуль — це багаторазовий будівельний блок. Замість того, щоб писати всі 500 рядків Terraform в одному файлі, ми розбиваємо їх на логічні групи (VPC, балансувальник, сервер, база даних). Кожен модуль має свою папку, власні входи та виходи. Це спрощує розуміння, тестування та повторне використання коду.

## Як задеплоїти (локально)

> ⚠️ **Важливо:** `terraform apply` створює реальні AWS ресурси і **коштує реальних грошей**. Завжди спочатку запускай `terraform plan`, щоб переглянути що буде створено.

### Крок 1 — Клонуй репозиторій

```bash
git clone https://github.com/samarets-vlad/aws-terraform-infra.git
cd aws-terraform-infra
```

### Крок 2 — Налаштуй AWS credentials

```bash
aws configure
# Введи: AWS Access Key ID, Secret Access Key, Region (наприклад eu-central-1)
```

### Крок 3 — Створи файл зі змінними

```bash
cp terraform.tfvars.example terraform.tfvars
# Відкрий terraform.tfvars і заповни свої значення
```

Мінімально необхідні значення:

```hcl
project_name = "my-app"
environment  = "dev"
aws_region   = "eu-central-1"

vpc_cidr                  = "10.0.0.0/16"
availability_zones        = ["eu-central-1a", "eu-central-1b"]
public_subnet_cidrs       = ["10.0.1.0/24", "10.0.2.0/24"]
private_app_subnet_cidrs  = ["10.0.11.0/24", "10.0.12.0/24"]
private_db_subnet_cidrs   = ["10.0.21.0/24", "10.0.22.0/24"]

ami_id       = "ami-0a1b2c3d4e5f6"   # Специфічний для регіону — знайди в AWS Console
db_password  = "ЗМІНИ_МЕ_використай_надійний_пароль"
```

> 🔒 **Ніколи не комить `terraform.tfvars` в Git.** Він вже є в `.gitignore`. Там зберігається пароль від бази даних.

### Крок 4 — Ініціалізуй Terraform

```bash
terraform init
```

Завантажує плагін AWS провайдера (~40 МБ). Потрібно лише один раз або після змін у `versions.tf`.

### Крок 5 — Перегляд плану

```bash
terraform plan
```

Terraform покаже **точно** що він створить, змінить або знищить — нічого не роблячи насправді. Прочитай уважно перед тим, як продовжити.

### Крок 6 — Застосування

```bash
terraform plan -out=tfplan   # Зберегти план
terraform apply tfplan        # Застосувати саме цей план
```

### Крок 7 — Знищити (коли закінчив)

```bash
terraform destroy
```

Видаляє **всі** ресурси. Корисно для dev/test щоб уникнути постійних витрат.

> ⚠️ **Для продакшену:** `rds_deletion_protection = true` запобігає випадковому видаленню бази даних. Щоб запустити `destroy`, потрібно спочатку встановити це значення в `false`.

## Вхідні змінні

Всі змінні визначені у `variables.tf`. Скопіюй `terraform.tfvars.example` і заповни свої значення.

### Обов'язкові (без дефолту — потрібно вказати)

| Змінна | Тип | Опис |
|--------|-----|------|
| `vpc_cidr` | string | Діапазон мережі VPC, наприклад `10.0.0.0/16` |
| `public_subnet_cidrs` | list | CIDR блоки публічних підмереж, по одному на AZ |
| `private_app_subnet_cidrs` | list | CIDR блоки приватних підмереж для застосунку |
| `private_db_subnet_cidrs` | list | CIDR блоки підмереж для БД |
| `availability_zones` | list | Зони доступності, наприклад `["eu-central-1a", "eu-central-1b"]` |
| `ami_id` | string | ID AMI для EC2 (специфічний для регіону — знайди в AWS Console) |
| `db_password` | string | Мастер пароль PostgreSQL — **ніколи не хардкодити** |

### Необов'язкові (мають розумні дефолти)

| Змінна | Дефолт | Опис |
|--------|--------|------|
| `aws_region` | `eu-central-1` | AWS регіон для деплою |
| `project_name` | `portfolio-app` | Префікс для всіх назв ресурсів |
| `environment` | `dev` | Має бути `dev`, `stage` або `prod` |
| `instance_type` | `t3.micro` | Тип EC2 інстансу |
| `app_port` | `8080` | Порт, на якому слухає застосунок |
| `db_instance_class` | `db.t3.micro` | Тип RDS інстансу |
| `db_name` | `appdb` | Назва бази даних PostgreSQL |
| `db_username` | `appuser` | Мастер юзернейм PostgreSQL |
| `rds_deletion_protection` | `true` | Захист від випадкового видалення БД |
| `rds_skip_final_snapshot` | `false` | Якщо false — знімок робиться перед видаленням |
| `rds_backup_retention_period` | `7` | Днів зберігати автоматичні бекапи |
| `ssh_allowed_cidrs` | `[]` | CIDRs з яких дозволений SSH. Порожньо = SSH вимкнено |
| `ssh_key_name` | `null` | Назва EC2 key pair. Null = без key pair |

## CI/CD Пайплайн

Кожен push та кожен Pull Request автоматично запускає пайплайн, визначений у `.github/workflows/terraform.yml`.

```
Push або Pull Request до main
          │
          ▼
    ┌─────────────┐
    │  1. Validate │  terraform fmt, init (без backend), validate, tflint
    └──────┬──────┘
           │ пройшло
           ▼
    ┌─────────────┐
    │  2. Security │  tfsec — сканування на AWS місконфіги
    └─────────────┘

  ✅ Якщо обидва пройшли → PR можна змерджити
  ❌ Якщо щось не пройшло → PR заблоковано
```

### Що перевіряє кожен крок:

| Крок | Інструмент | Що перевіряє |
|------|------------|---------------|
| Форматування | `terraform fmt -check` | Код правильно відформатований (відступи, вирівнювання) |
| Init | `terraform init -backend=false` | Залежності завантажуються правильно |
| Validate | `terraform validate` | Синтаксис правильний, немає невизначених змінних |
| Lint | `tflint` | Кращі практики: закріплені версії, задокументовані змінні тощо |
| Безпека | `tfsec` | Немає відкритих security груп, шифрування увімкнено, IMDSv2 увімкнено тощо |

> 💡 **У CI не використовуються AWS credentials.** Всі перевірки — статичний аналіз коду без підключення до AWS. Це означає **нульову вартість** запуску CI.

> ⚠️ **`terraform plan` та `terraform apply` навмисно не входять до CI.** Запускай їх вручну з локальної машини коли реально хочеш деплоїти.

## Захист безпеки

Цей проєкт реалізує кращі практики безпеки AWS «з коробки»:

| Що | Чому це важливо |
|----|------------------|
| **IMDSv2 обов'язковий на EC2** | Запобігає SSRF атакам, які крадуть credentials інстансу |
| **EBS томи зашифровані** | Дані в спокої зашифровані AES-256 |
| **RDS сховище зашифроване** | Дані бази даних зашифровані в спокої |
| **Захист від видалення RDS** | Запобігає випадковому знищенню бази даних |
| **Фінальний snapshot RDS** | Резервна копія автоматично робиться перед будь-яким видаленням |
| **S3 публічний доступ заблоковано** | Всі 4 прапори блокування публічного доступу встановлені в `true` |
| **S3 версіонування увімкнено** | Можна відновити видалені або перезаписані файли |
| **S3 серверне шифрування** | Файли в S3 зашифровані |
| **SSH вимкнено за замовчуванням** | `ssh_allowed_cidrs = []` — правило SSH відсутнє якщо явно не вказати |
| **БД у приватній підмережі** | RDS не має доступу до інтернету — доступна тільки з серверів застосунку |
| **Секрети не в коді** | `db_password` позначений як `sensitive`, ніколи не зберігається в Git |
| **Версія провайдера закріплена** | `aws ~> 5.98` — без несподіваних критичних змін |
| **Terraform >= 1.11.0 обов'язковий** | Мінімальна версія примусово вказана у всіх модулях |

## State файл

Terraform повинен зберігати «state файл» — запис того, яка інфраструктура існує зараз. Цей файл потрібно зберігати віддалено (не на ноутбуці), щоб команда могла разом працювати і щоб він не загубився.

Цей проєкт налаштований використовувати **AWS S3 + DynamoDB** для зберігання remote state (дивись `backend.tf`).

Перед першим запуском `terraform init` потрібно **вручну створити**:
- S3 бакет (з увімкненим версіонуванням та шифруванням)
- DynamoDB таблицю (для блокування state — запобігає запуску `apply` двома людьми одночасно)

Потім оновити `backend.tf` із назвою свого бакету і таблиці.

> Це разовий крок налаштування. Після цього весь state управляється автоматично.

## Відмінності між оточеннями

Використовуй змінну `environment` для контролю поведінки в кожному оточенні:

| Налаштування | `dev` | `stage` | `prod` |
|-------------|-------|---------|--------|
| `rds_deletion_protection` | `false` | `true` | `true` |
| `rds_skip_final_snapshot` | `true` | `false` | `false` |
| `rds_backup_retention_period` | `1` | `7` | `14` |
| `instance_type` | `t3.micro` | `t3.small` | `t3.medium` |
| `ssh_allowed_cidrs` | твій IP | твій IP | `[]` (вимкнено) |
